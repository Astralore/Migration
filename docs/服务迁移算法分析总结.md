## 一、整体架构概览

整个代码库围绕 **边缘环境下的服务迁移** 问题构建，核心流程为：

```
用户移动轨迹数据 → 轨迹预测模型 → 迁移决策算法 → 评估（迁移次数 M + 违规次数 V）
```

代码分为三大模块：

| 模块 | 路径 | 说明 |
|------|------|------|
| 轨迹预测 | `SpatioFormer_Code/` + 预测器类 | 预测用户未来位置，为迁移决策提供输入 |

## 二、核心问题建模

### 2.1 场景描述

出租车（代表移动用户）在北京区域移动，由多个边缘服务器提供计算服务。当用户远离当前服务的边缘服务器时，需要做出 **是否迁移** 以及 **迁移到哪个服务器** 的决策。

### 2.2 关键参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 服务半径阈值 | 15 km | 距离超过此值视为违规（violation） |
| 极端距离阈值 | 30 km | 无条件触发迁移 |
| 候选服务器数 k | 3（部分算法为 5） | 从最近的 k 个边缘服务器中选择 |
| 预测步长 | 10 步（评估前 8 步） | 轨迹预测的前瞻窗口 |
| 评估指标 | `Score = M + 0.5V` | M=迁移次数，V=违规次数，越低越好 |
| 仿真采样 | 8787 个时间步 | 所有方法统一使用相同的时间步子集 |

### 2.3 迁移触发条件（Context-Aware）

所有算法共享统一的迁移触发函数 `check_migration_criteria`：

```python
def check_migration_criteria(row, current_dist):
    if current_dist <= 15.0:
        return False  # 距离在范围内，不需要迁移

    # 上下文感知：融合用户健康数据作为附加触发条件
    health_risk = (
        age > 75 or
        physical_activity == 'high' or
        cvd_risk_score > 70 or
        hypertension == 'yes' or
        diabetes == 'yes' or
        current_dist > 30.0  # 极端距离无条件触发
    )
    return health_risk
```

**核心设计**：不仅考虑距离，还融合了用户的健康数据（年龄、体力活动水平、心血管风险评分、高血压、糖尿病）作为迁移的附加触发因素。这体现了 **上下文感知（Context-Aware）** 的设计理念。

### 2.4 通用仿真框架伪代码

```
对于每个时间步 t:
    对于每辆出租车 taxi:
        如果 taxi 未分配服务器:
            分配到最近的服务器
            continue
        
        计算 taxi 到当前服务器的距离 d
        
        如果 d > 15km:
            violations += 1
            如果满足迁移条件(health_risk):
                获取 k 个候选服务器
                [各算法在此处做不同的决策]
                如果决策结果 ≠ 当前服务器:
                    migrations += 1
                    更新分配
```

## 三、轨迹预测模型

轨迹预测为迁移决策算法提供 **未来位置预测**，使算法能做前瞻性决策。

### 3.1 SimpleTrajectoryPredictor（恒速预测器）

**文件**：`Context_Integratoin.py` 第 29-60 行

**原理**：对每辆出租车计算历史经纬度的平均变化量（平均速度），用恒速外推预测未来位置。

```python
class SimpleTrajectoryPredictor:
    def fit(self, df):
        # 为每辆出租车计算 (mean_dx, mean_dy)
        for taxi_id in df['taxi_id'].unique():
            dx = np.diff(lons)  # 经度方向平均步长
            dy = np.diff(lats)  # 纬度方向平均步长
            self.velocity_factors[taxi_id] = (np.mean(dx), np.mean(dy))

    def predict_future(self, current_lon, current_lat, taxi_id, steps):
        # 从当前位置开始，每步叠加平均速度
        for _ in range(steps):
            lon += dx; lat += dy
            future.append((lon, lat))
        return future
```

**用途**：被 Heuristic、SA、ILP、Hybrid 等所有前瞻性算法调用，评估候选服务器在未来的违规情况。

### 3.2 SimpleSpatialTransformer（空间变换预测器）

**文件**：`Predictive_Analysis_Functions.py` 第 28-140 行

**原理**：与 SimpleTrajectoryPredictor 相同（恒速模型），但增加了完整的评估框架和模型持久化功能。

**评估指标**：
- RMSE / MAE（经度和纬度）
- Haversine 距离误差（均值、中位数、标准差）
- 多阈值精度（1km / 2km / 5km 内的预测准确率）

### 3.3 深度学习轨迹预测模型（SpatioFormer_Code/）

提供了 7 种深度学习模型用于轨迹预测评估：

| 模型 | 文件 | 核心结构 | 特点 |
|------|------|----------|------|
| CNN | `CNN.py` | 1D Conv + Pooling | 捕捉局部空间模式 |
| LSTM | `LSTM.py` | 双层 LSTM | 长程时序依赖 + DBSCAN轨迹聚类 |
| GRU | `GRU.py` | 双层 GRU + BatchNorm | LSTM的轻量替代 |
| CNN-LSTM | `CNN_LSTM.py` | CNN特征提取 + LSTM时序 | 空间+时间联合建模 |
| STT-FNN | `STT_FNN.py` | Transformer Encoder + 因果mask | 注意力机制 |
| AgentFormer | `AgentFormer.py` | Agent-Aware自注意力 | 多智能体感知 |
| **SpatioFormer** | `SpatioFormer(OUR).py` | 自定义Multi-Head Attention + Transformer | **本文提出方法** |

**共同流程**：读取出租车 GPS 数据 → 轨迹切分/聚类(DBSCAN)或滑窗 → 标准化 → PyTorch Dataset → 模型训练 → Haversine 误差评估

> **当前状态**：这些深度学习模型 **未与迁移决策算法集成**，仅单独做轨迹预测评估。迁移决策使用的是简单的恒速预测器。

## 四、8 种迁移决策算法详解

### 4.1 Reactive Baseline（反应式基线）

**文件**：`Context_Integratoin.py` → `run_reactive_fair()`

**核心思想**：最简单的策略——只在违规发生时才反应，迁移到当前最近的服务器。

**决策流程**：

```
当 current_dist > 15km 且满足健康条件:
    找到当前位置最近的 1 个服务器
    直接迁移过去
```

**特点**：
- 无预测、无优化，纯粹的贪心反应策略
- 作为性能下界基准（baseline）
- 时间复杂度 O(N)，N 为服务器数量

---

### 4.2 Heuristic（启发式前瞻）

**文件**：`Context_Integratoin.py` → `run_heuristic_fair()` / `Heuristic_Migratoins.py`

**核心思想**：利用轨迹预测，在 k 个候选服务器中选择 **未来违规次数最少** 的一个。

**决策流程**：

```
当 current_dist > 15km 且满足健康条件:
    获取 k=3 个最近候选服务器
    预测未来 10 步位置
    
    对于每个候选服务器 j:
        统计前 8 步中 距离j > 15km 的次数 → future_violations_j
    
    选择 future_violations 最小的候选服务器
```

**关键参数**：

| 参数 | 值 |
|------|-----|
| 候选数 k | 3 |
| 预测步数 | 10 |
| 评估步数 | 前 8 步 |

**特点**：
- 确定性算法，穷举所有候选的未来违规计数，选最优
- 利用了轨迹预测做前瞻决策
- 缺点：只考虑违规计数，不考虑迁移开销的权衡

---

### 4.3 Simulated Annealing（模拟退火）

**文件**：`Context_Integratoin.py` → `run_sa_fair()` / `SA_Migration.py`

**核心思想**：用模拟退火在候选服务器中搜索，以一定概率接受较差解来跳出局部最优。

**决策流程**：

```
当 current_dist > 15km 且满足健康条件:
    获取 k=3 个候选，预测 10 步未来位置
    初始化: T=50.0, cooling=0.99, best=当前服务器
    
    重复 15 次:
        随机选一个候选服务器 c
        计算 c 的未来违规次数 v_c
        delta = v_c - best_violations
        
        若 delta < 0: 接受 c（更好的解）
        否则: 以概率 exp(-delta/T) 接受（可能跳出局部最优）
        
        T = T × 0.99  # 降温
        若 T < 0.001: 终止
```

**关键参数**：

| 参数 | 值 |
|------|-----|
| 初始温度 T₀ | 50.0 |
| 冷却率 α | 0.99 |
| 最大迭代次数 | 15 |
| 终止温度 | 0.001 |

**代码实现注意事项**：

当前代码在接受概率判断 `if delta < 0 or random.random() < exp(-delta/T)` 之后，又嵌套了 `if future_violations < best_future_violations` 才实际更新 best。这意味着 **实际上永远不会接受较差解**，SA 退化为纯随机贪心搜索。标准 SA 实现中，接受概率通过后应直接更新当前解（而非 best 解），best 解始终保留最优。

---

### 4.4 Q-Learning（表格型 Q 学习）

**文件**：`Context_Integratoin.py` → `run_ql_fair()` / `Q-Learning.py`

**核心思想**：将迁移决策建模为强化学习（RL）问题，用 Q-table 学习状态-动作的价值函数。

**MDP 建模**：

| 要素 | 定义 |
|------|------|
| **状态 S** | `(lat_bucket, lon_bucket, dist_bucket)` — 离散化的位置和距离 |
| **动作 A** | `{保持当前, 选最近, 选次近, 选第三近}` — 4 个离散动作 |
| **奖励 R** | 在15km内 → +8；超出 → -12；发生迁移 → -2 |
| **Q表** | `defaultdict(lambda: np.zeros(4))` |

**状态离散化**：

```python
state = (
    round(current_lat / 90.0 * 10),    # 纬度桶（约9°精度）
    round(current_lon / 180.0 * 10),   # 经度桶（约18°精度）
    min(int(current_dist / 5), 5)      # 距离桶（0-25km，5km间隔）
)
```

**Q 值更新（标准 Q-learning）**：

$$Q(s,a) \leftarrow Q(s,a) + \alpha \left[ r + \gamma \cdot \max_{a'} Q(s',a') - Q(s,a) \right]$$

**关键参数**：

| 参数 | 值 |
|------|-----|
| 学习率 α | 0.1 |
| 折扣因子 γ | 0.9 |
| 探索率 ε | 0.2（固定） |

**特点**：
- 在线学习，在运行过程中逐步改善策略
- 不使用轨迹预测信息
- 状态空间离散化较粗糙
- ε 固定不衰减，始终保持探索

### 4.5 DQN（深度 Q 网络）

**文件**：`Context_Integratoin.py` → `run_dqn_fair()` / `DQN_Migration.py`

**核心思想**：用神经网络替代 Q-table，处理连续状态空间，引入 Experience Replay 和 Target Network 稳定训练。

**网络结构**（`SimpleDQN`）：

```
Input(10) → Linear(64) → ReLU → Linear(64) → ReLU → Linear(4) → Q-values
```

**状态向量**（10 维连续特征）：

```python
state = [
    current_lat / 90.0,           # 归一化纬度
    current_lon / 180.0,          # 归一化经度
    min(current_dist / 50.0, 1.0),# 归一化距离
    timestamp.hour / 24.0,        # 小时特征
    timestamp.weekday() / 7.0,    # 星期特征
    len(candidates) / 10.0,       # 候选数量
    0.0, 0.0, 0.0,               # 预留占位特征
    1.0 if current_dist > 15.0 else 0.0  # 是否违规
]
```

**训练机制**：

| 组件 | 配置 |
|------|------|
| Experience Replay | 容量 2000，batch=32 |
| Target Network | 每 50 步同步一次 |
| ε-greedy | ε: 0.3 → 0.01，衰减率 0.995 |
| 优化器 | Adam, lr=0.001 |
| 折扣因子 γ | 0.95 |
| 损失函数 | MSE Loss |

**奖励结构**：

| 条件 | 奖励 |
|------|------|
| 新距离 ≤ 15km | +8 |
| 新距离 > 15km | -12 |
| 发生迁移 | -2 |

**简化说明**：代码中 `memory.append((state, action, reward, state, False))` 将 next_state 直接设为当前 state，这是一个简化处理。

### 4.6 Hybrid SA-Q-Learning（SA 基础 + Q-Learning 精炼）

**文件**：`Context_Integratoin.py` → `run_hybrid_sa_ql_fair()` / `SA-Qlearning.py`

**核心思想**：两层决策架构。第一层用增强版 SA 搜索最佳服务器并给出建议；第二层用 Q-Learning 在多个候选方案中做最终选择。

**两个版本差异**：

| 特性 | Context_Integratoin.py 版本 | SA-Qlearning.py 版本 |
|------|---------------------------|---------------------|
| Q-Learning 动作数 | 4（接受SA、最近、次近、第三近） | 3（保持当前、SA建议、历史最优） |
| 状态表示 | `(dist_cat, time_period, sa_quality)` | `(dist_cat, time_cat, recent_violations)` |
| 是否维护历史记录 | 否 | 是（performance + migration_history） |

**增强版 SA 参数**（区别于纯 SA）：

| 参数 | 纯 SA | 混合版 SA |
|------|-------|----------|
| 冷却率 | 0.99 | 0.85（更快冷却） |
| 迭代次数 | 15 | 20（更充分搜索） |

**SA-Qlearning.py 版本独特设计**：

1. **历史表现追踪**：维护 `taxi_server_performance[(taxi_id, server_id)]`，记录每对出租车-服务器的成功率
2. **迁移历史记录**：维护 `taxi_migration_history[taxi_id]`，保留最近 5 条记录
3. **Q-Learning 状态**：包含 `recent_violations`（最近违规次数），使策略能适应违规频率变化

**决策流程**（SA-Qlearning.py 版本）：

```
1. 运行增强SA → 得到 sa_best_server
2. 查询历史表现 → 得到 historical_best_server
3. Q-Learning ε-greedy 选择动作:
   - action=0: 保持当前服务器
   - action=1: 使用 SA 推荐
   - action=2: 使用历史最优
4. 执行选择，计算奖励，更新Q表
5. 更新历史表现记录
```

### 4.7 RL-Refined SA（DQN 作为 SA 的评判者）

**文件**：`Context_Integratoin.py` → `run_rl_refined_sa_fair()`

**核心思想**：最复杂的混合算法。SA 产出迁移建议后，由一个具有更丰富状态表示的 DQN 网络决定最终方案。

**DQN 网络**（`SimpleRefinementDQN`）：

```
Input(12) → Linear(64) → ReLU → Linear(64) → ReLU → Linear(3) → Q-values
```

**状态向量**（12 维，比纯 DQN 更丰富）：

```python
state = [
    current_lat / 90.0,              # 1. 归一化纬度
    current_lon / 180.0,             # 2. 归一化经度
    min(current_dist / 50.0, 1.0),   # 3. 归一化当前距离
    sa_proposed_lat / 90.0,          # 4. SA建议服务器纬度
    sa_proposed_lon / 180.0,         # 5. SA建议服务器经度
    sa_predicted_violations / 10.0,  # 6. SA建议的预期违规率
    timestamp.hour / 24.0,           # 7. 小时
    timestamp.weekday() / 7.0,       # 8. 星期
    (lat - sa_lat) / 10.0,          # 9. 相对纬度差
    (lon - sa_lon) / 10.0,          # 10. 相对经度差
    dist_to_sa / 50.0,              # 11. 到SA选择的距离
    len(candidates) / 5.0            # 12. 候选数量
]
```

**3 个动作**：

| 动作 | 说明 |
|------|------|
| action=0 | 接受 SA 的推荐 |
| action=1 | 选择绝对最近的服务器 |
| action=2 | 选择历史表现最好的服务器（带置信度加成） |

**奖励结构**（比其他算法更强烈的信号）：

| 条件 | 奖励 |
|------|------|
| 新距离 ≤ 15km | +12.0 |
| 新距离 > 15km | -15.0 |
| 发生迁移 | -2.0 |

**训练配置**：

| 组件 | 配置 |
|------|------|
| Replay Buffer | 容量 2000，最小 200 条后开始训练 |
| Batch Size | 32 |
| Target Network | 每 50 步同步 |
| ε 衰减 | 0.3 → 0.05，衰减率 0.997 |
| 学习率 | 0.001 |
| 折扣因子 | 0.9 |

### 4.8 ILP（整数线性规划）

**文件**：`Context_Integratoin.py` → `run_ilp_fair()` / `ILP_Migration.py`

**核心思想**：将每个迁移决策建模为精确的数学优化问题，用 ILP 求解器（PuLP + CBC）求全局最优。

**数学模型**：

$$\min \sum_{j \in \text{candidates}} \left( w_v \cdot \text{violations}_j \cdot x_j + w_m \cdot \text{migration\_cost}_j \cdot x_j \right)$$

$$\text{s.t.} \quad \sum_{j} x_j = 1 \quad \text{（必须且只能选一个服务器）}$$

$$x_j \in \{0, 1\} \quad \text{（二值决策变量）}$$

**参数定义**：

| 符号 | 含义 | 值 |
|------|------|-----|
| $x_j$ | 是否选择候选服务器 j | 0 或 1 |
| $\text{violations}_j$ | 候选 j 的未来 5 步违规次数 | 0-5 |
| $\text{migration\_cost}_j$ | 迁移成本 | 0（留原地）或 1（迁移） |
| $w_v$ | 违规权重 | 10.0 |
| $w_m$ | 迁移权重 | 1.0 |

**实现细节**：
- 使用 PuLP 库建模，CBC 求解器求解
- 每个决策限时 1 秒
- 求解失败时回退到启发式方法
- PuLP 不可用时回退到纯启发式

**特点**：
- 理论上在给定预测信息下找到最优解
- 每步都要调用求解器，计算开销最大
- 优化范围仅限单个用户的单步决策，非全局联合优化

### 5.3 独立 SA 实现

```python
def simulated_annealing(initial_servers, current_server, lat, lon, candidates,
                        servers_df, temp=100, cooling_rate=0.95, max_iter=50):
    # 目标函数：最小化到所选服务器的总距离
    def compute_cost(assignment):
        return sum(haversine_distance(lat, lon, s_lat, s_lon) for s in assignment)

    for _ in range(max_iter):
        neighbor = random.choice(candidates)     # 随机邻域
        delta = neighbor_cost - current_cost
        if delta < 0 or random.random() < exp(-delta/T):
            current = neighbor                    # 标准SA接受准则
            if current_cost < best_cost:
                best = current
        T *= cooling_rate
    return best
```

> 这个 SA 实现与 `Context_Integratoin.py` 中的版本不同：目标函数是**当前距离最小化**而非**未来违规最小化**。

## 六、评估框架

### 6.1 统一评估指标

$$\text{Score} = M + 0.5 \times V$$

- **M (Migrations)**：总迁移次数，代表迁移开销（网络带宽、服务中断等）
- **V (Violations)**：总违规次数，代表服务质量损失（用户超出服务范围的时步数）
- **目标**：Score 越低越好

权重 0.5 表明：1 次迁移的代价 等价于 2 次违规的代价。

### 6.2 公平对比保障

所有方法在完全相同的条件下运行：
- 相同的时间步子集（8787 个采样点）
- 相同的迁移触发条件（`check_migration_criteria`）
- 相同的数据分组方式（`df.groupby('date_time')`）
- 相同的初始分配策略（首次出现分配到最近服务器）
- 相同的违规统计逻辑

## 七、各算法对比总结

| 算法 | 类型 | 是否用预测 | 决策机制 | 学习能力 | 时间复杂度 | 关键超参 |
|------|------|-----------|---------|---------|-----------|---------|
| Reactive | 贪心 | 否 | 最近服务器 | 无 | O(N) | 无 |
| Heuristic | 确定性搜索 | 是 | 穷举k个候选 | 无 | O(k·T) | k=3, T=8步 |
| SA | 元启发式 | 是 | 随机搜索+退火 | 无 | O(iter·T) | T₀=50, α=0.99, iter=15 |
| Q-Learning | 表格RL | 否 | ε-贪心+Q表 | 在线学习 | O(1)/步 | α=0.1, γ=0.9, ε=0.2 |
| DQN | 深度RL | 否 | 神经网络Q值 | 在线学习 | O(batch·forward) | lr=0.001, γ=0.95, batch=32 |
| Hybrid SA-QL | 混合(SA+RL) | 是 | SA建议+Q精炼 | 在线学习 | O(SA+Q更新) | 双层参数组合 |
| RL-Refined SA | 深度混合 | 是 | SA建议+DQN评判 | 在线学习 | O(SA+DQN训练) | 12维状态, 3动作 |
| ILP | 精确优化 | 是 | 数学规划求解器 | 无 | O(求解器) | w_v=10, w_m=1 |

### 算法分类视角

```
                    ┌─ Reactive（无预测基线）
                    │
        ┌─ 无学习 ──┼─ Heuristic（确定性前瞻）
        │           ├─ SA（随机化前瞻）
        │           └─ ILP（精确优化前瞻）
算法 ───┤
        │           ┌─ Q-Learning（表格型，无预测）
        └─ 有学习 ──┼─ DQN（深度型，无预测）
                    ├─ Hybrid SA-QL（SA前瞻 + Q精炼）
                    └─ RL-Refined SA（SA前瞻 + DQN评判）
```

## 八、面向微服务迁移改造的关键差异分析

当前代码是 **单体服务迁移**（一辆车 ↔ 一个边缘服务器），改造为 **微服务迁移** 需要关注以下差异：

### 8.1 迁移对象的变化

| 维度 | 当前（单体服务） | 微服务 |
|------|----------------|--------|
| 迁移单元 | 用户服务整体 | DAG 中各个微服务实例 |
| 决策维度 | 选 1 个目标服务器 | 为每个微服务选放置节点 |
| 决策空间 | O(k) | O(k^n)，n=微服务数量 |

### 8.2 迁移成本模型

当前代码中迁移成本是**固定惩罚值**（-1 或 -2），微服务迁移需要考虑：

- `image_mb`：容器镜像传输大小
- `state_mb`：有状态服务的状态数据迁移量
- `is_stateful`：有状态服务的迁移代价远高于无状态服务
- **服务间通信开销**：DAG 边权重代表调用频率，跨节点部署会引入额外网络延迟

### 8.3 新增约束

| 约束类型 | 当前代码 | 微服务场景需要 |
|---------|---------|--------------|
| 资源容量 | 无 | 每个边缘节点的 CPU/内存/带宽上限 |
| 服务依赖 | 无 | DAG 拓扑约束、共置偏好 |
| 延迟要求 | 仅距离阈值 | 端到端调用链延迟 SLA |
| 并发实例 | 1:1 映射 | 多副本、负载均衡 |

### 8.4 各算法改造要点

| 算法 | 微服务改造方向 |
|------|--------------|
| Reactive | 可保留为基线，但需支持多实例分别迁移 |
| Heuristic | 违规计数需改为综合延迟指标，候选评估需考虑 DAG 共置 |
| SA | 邻域操作需改为"移动某个微服务到另一节点"，目标函数需纳入通信开销 |
| Q-Learning | 状态空间需加入 DAG 拓扑特征、实例负载等 |
| DQN | 状态向量需扩展微服务相关特征，动作空间需重新设计 |
| Hybrid | 两层架构可保留，SA 和 QL 各自需适配微服务 |
| RL-Refined SA | 12维状态需扩展为包含微服务信息的更高维表示 |
| ILP | 需大幅重构：增加资源约束、依赖约束、延迟约束 |

### 8.5 你已完成的工作

`microservice_knowledge_base.py` 中定义了 3 种工作负载类型的微服务 DAG：

| DAG 名称 | 出现概率 | 节点数 | 特点 |
|---------|---------|--------|------|
| Data_Heavy_DAG | 40% | 6 | 大流量边（13937），有状态节点 |
| Compute_Heavy_DAG | 35% | 4 | 超大流量边（22707），全无状态 |
| IoT_Lightweight_DAG | 25% | 3 | 轻量级，全有状态（state_mb=10） |

这些 DAG 数据目前 **尚未被任何迁移算法引用**，后续需要将其集成到迁移决策流程中。
